# BNI 마포홍보관 QA 테스트 계획서

## 1. 테스트 전략

### 1.1 목표
- 10개 페이지, 16개 컴포넌트에 대한 포괄적 E2E 테스트 커버리지 확보
- 인증/권한, 상품 CRUD, 주문 플로우, 관리자 기능의 정상 동작 검증
- 멤버리스트 기반 가입승인 메커니즘의 현재 상태 분석 및 개선 방향 제시
- 모바일/데스크톱 반응형 동작 검증

### 1.2 범위
| 구분 | 대상 |
|------|------|
| 페이지 | Login, Register, Profile, Home, MyProducts, EditProduct, Admin, Guide, DreamReferral, Chapters |
| 컴포넌트 | ProductForm, ProductCard, ProductModal, OrderForm, CategoryFilter, SearchBar, AddressSearch, MemberDirectory, DreamForm, DreamCard, DreamDetailModal, ChapterCard, Header, BottomNav, Footer, Layout |
| 훅/스토어 | useAuth, useProducts, useOrders, useChapters, useDreamReferrals, useMembers |
| Edge Functions | send-order-email, update-members, update-chapters |

### 1.3 도구
- **E2E 테스트**: Playwright (chromium, 새 브라우저 인스턴스)
- **배포 URL**: https://bni-orcin.vercel.app
- **테스트 계정**: info_nomu@naver.com / 0123456
- **주의사항**: networkidle 사용 금지 -> domcontentloaded 사용

### 1.4 접근방식
- 라이브 배포 URL 대상 E2E 테스트 (Supabase 백엔드 직접 연결)
- 각 테스트는 독립적으로 실행 가능해야 함
- 데이터 변경 테스트는 테스트 후 원상복구 (cleanup)
- 스크린샷 캡처로 시각적 검증 보완

### 1.5 테스터 분배
| 테스터 | 담당 영역 | 페이지/컴포넌트 |
|--------|-----------|-----------------|
| qa-auth | 인증/프로필 | Login, Register, Profile, useAuth |
| qa-product | 상품/주문 | Home, MyProducts, EditProduct, OrderForm, ProductForm, ProductModal, SearchBar, CategoryFilter |
| qa-feature | 부가기능 | Admin, Guide, DreamReferral, Chapters, MemberDirectory, DreamForm, DreamDetailModal, ChapterCard |

## 3. 멤버리스트 가입승인 검증

### 3.1 현재 구현 분석

**signUp 로직** (`src/hooks/useAuth.js:85-136`)에서 확인된 사항:

1. **멤버리스트 검증 없음**: `signUp` 함수는 Supabase auth.signUp을 직접 호출하며, `chapter_members` 테이블과 대조하는 로직이 없음
2. **현재 플로우**: 이메일/비밀번호 + 프로필 데이터 -> Supabase Auth 가입 -> profiles 테이블 upsert
3. **챕터 선택은 자유**: 드롭다운에서 아무 챕터나 선택 가능, BNI 멤버 여부 확인 없음
4. **관리자 측 멤버 관리**: Admin 페이지에서 `update-members` Edge Function으로 BNI 사이트의 멤버 목록을 `chapter_members` 테이블에 동기화
5. **탈퇴 처리 메커니즘**: `chapter_members`에서 사라진 멤버의 `profiles.status`를 'suspended'로 변경, 로그인 차단

### 3.2 검증 시나리오

### 3.3 개선 권고사항

**현재 상태 (AS-IS)**:
- 누구나 이메일로 가입 가능 (BNI 멤버 여부 확인 없음)
- 가입 후 관리자가 수동으로 chapter_members와 매핑
- 탈퇴 처리는 멤버 동기화를 통해 자동 수행

**개선 방향 (TO-BE)** - 세 가지 옵션:

| 옵션 | 방식 | 장점 | 단점 |
|------|------|------|------|
| A. 가입 시 이름+챕터 매칭 | signUp 시 chapter_members 테이블에서 이름+챕터 조합 확인 | 자동 검증, 비멤버 차단 | 동명이인 이슈, 이름 표기 차이 |
| B. 관리자 승인제 | 가입 후 pending 상태 -> 관리자 승인 후 active | 정확한 검증 | 관리자 부담 증가 |
| C. 초대 코드 | 관리자가 챕터별 초대 코드 생성 -> 가입 시 코드 입력 | 간단한 구현 | 코드 유출 가능성 |

**권장 옵션**: **옵션 A** (이름+챕터 매칭)
- `chapter_members` 테이블의 `member_name`과 `chapter_name`으로 매칭
- 매칭 성공 시 자동으로 `profile_id` 연결
- 매칭 실패 시 "BNI 멤버로 확인되지 않습니다. 관리자에게 문의해주세요." 에러
